## PHASE 1: INVENTORY AND STRUCTURAL AUDIT

**1.1 — Directory walk**
Walk the entire project tree. Produce a complete annotated inventory of every file and directory with:

- File type and purpose
- Current status (production, placeholder, orphaned, under-construction)
- Whether the file is linked from any navigation or page
- Any file that exists in the repo but is not reachable from the live site navigation

**1.2 — Dead link detection**
Crawl every internal `href`, `src`, and `action` attribute in every HTML file. Flag every link that:

- Points to a path that does not exist in the repo
- Points to an anchor (`#id`) that does not exist on the target page
- Is a placeholder (`#`, `javascript:void(0)`, empty `href`)
- Points to a page serving `under-construction.html` content

Produce a dead link manifest. Fix every internal dead link you can fix by either: (a) creating the missing target page with appropriate placeholder content, or (b) removing the link and replacing it with a graceful in-progress notice.

**1.3 — File naming and path audit**
Check for:

- Inconsistent casing (e.g., `index.html` vs `Index.html`)
- Directory names with special characters that may cause routing issues (note: `found-ry` directory — verify this routes correctly)
- Any file referenced in HTML that uses a different case than its actual filename on disk
- Asset paths that are absolute when they should be relative, or vice versa

## PHASE 2: HTML QUALITY AUDIT — ALL PAGES

Run this audit against: `index.html`, `404.html`, `under-construction.html`, and every `index.html` inside subdirectories (about/, contact/, writings/, projects/, legal/, manifesto/, universe/, prompt-forge/, found-ry/, and all nested pages).

**2.1 — Document structure**
Each page must have:

- `<!DOCTYPE html>`
- `lang="en"` on `<html>`
- Proper `<head>` with `charset` meta first, then `viewport`, then title, then everything else
- A single `<h1>` per page (not zero, not two)
- Logical heading hierarchy (`h1` → `h2` → `h3`, no skipped levels)
- `<main>` element wrapping page content
- `<header>`, `<nav>`, `<footer>` as semantic landmarks
- Skip navigation link that actually works (`<a href="#main" class="skip-link">`)

Fix every violation. Do not leave a single page with broken document structure.

**2.2 — Title tag and meta description audit**
Each page must have:

- A unique `<title>` tag using the format: `[Page Name] | OverKill Hill P³™`
- A unique `<meta name="description">` between 120–160 characters, written in the OverKill Hill brand voice (confident, precise, no filler phrases)
- No duplicate titles across pages
- No title tags that are generic placeholders (“Untitled”, “Page”, or identical to another page)

Write these for every page. Do not use the same description twice.

**2.3 — Open Graph and Twitter Card meta**
Every page must have the following fully populated Open Graph block:

```html
<meta property="og:title" content="[Page Title]">
<meta property="og:description" content="[Unique description]">
<meta property="og:type" content="website"> <!-- or "article" for writing pages -->
<meta property="og:url" content="https://overkillhill.com/[path]/">
<meta property="og:image" content="https://overkillhill.com/assets/img/[appropriate image]">
<meta property="og:site_name" content="OverKill Hill P³™">
```

And the Twitter Card block:

```html
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:site" content="@OverKillHillP3">
<meta name="twitter:title" content="[Page Title]">
<meta name="twitter:description" content="[Unique description]">
<meta name="twitter:image" content="https://overkillhill.com/assets/img/[appropriate image]">
```

Article pages (writings/) should use `og:type` = `article` and include:

```html
<meta property="article:author" content="Jamie Hill">
<meta property="article:published_time" content="[ISO date]">
```

Implement this on every page. Identify which images are most appropriate as og:image for each page based on what’s available in assets/img/.

**2.4 — Canonical URLs**
Every page must have:

```html
<link rel="canonical" href="https://overkillhill.com/[path]/">
```

Trailing slash should be consistent with the site’s routing convention. Pick one convention and apply it everywhere.

**2.5 — Structured data (JSON-LD)**

The homepage needs `WebSite` and `Organization` schema:

```json
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "OverKill Hill P³™",
  "url": "https://overkillhill.com",

    "https://www.youtube.com/@OverKillHillP3",

  "contactPoint": {
    "@type": "ContactPoint",

  }
```

Writing pages (articles) need `Article` schema with `headline`, `author`, `datePublished`, `image`, `publisher`.

The Manifesto page should use `WebPage` schema.

Implement all of these.

## PHASE 4: PERFORMANCE AUDIT AND OPTIMIZATION

**4.1 — Image audit**
Inventory every image in assets/img/ and every `<img>` tag in HTML files.

For each image, check:

- Does it have a non-empty, descriptive `alt` attribute? (Not “image”, not empty, not the filename)
- Does it have explicit `width` and `height` attributes to prevent layout shift?
- Is it served in a modern format (WebP preferred over PNG/JPG where feasible)?
- Is any image being loaded that’s larger than 2x the maximum display size?
- Are hero/above-fold images marked with `loading="eager"` and below-fold images with `loading="lazy"`?

Fix every alt text that is generic or missing. Add width/height where absent. Add `loading="lazy"` to all below-fold images. Add `loading="eager"` and `fetchpriority="high"` to the primary hero image on each page.

**4.2 — CSS audit**

- Check for duplicate CSS rules across stylesheets
- Check for unused CSS selectors (selectors that target elements that don’t exist in any HTML page)
- Check that CSS custom properties (variables) are used consistently and not hardcoded colors scattered throughout
- Check that all color values use the site’s brand palette (identify the primary palette from what’s in use and document it in a comment block at the top of the main stylesheet)
- Check for render-blocking CSS that isn’t critical and could be deferred
- Verify the font stack is consistent across all pages
- Check for any `!important` declarations that could be eliminated

Produce a CSS remediation plan and implement every fix that doesn’t risk visual regression.

**4.3 — JavaScript audit**
The site currently has only 1.6% JavaScript. Audit what exists:

- What does each JS file do?
- Are any scripts included that are not used?
- Are scripts loaded with `defer` or `async` where appropriate?
- Is the mobile navigation hamburger implemented in JS? Does it work correctly? Is the “Toggle navigation” label visible as raw text anywhere in the rendered output? If so, fix the button so its text is visually hidden but accessible to screen readers using the proper technique (`visually-hidden` class or `aria-label`).

Implement the following JavaScript improvements if not already present:

- Smooth scroll behavior for all anchor links (`scroll-behavior: smooth` in CSS, or JS fallback)
- Active nav link highlighting based on current page URL
- Mobile nav: ensure tap outside the menu closes it
- Any inline `onclick` handlers should be moved to proper event listeners in external JS files

**4.4 — Resource loading order**
Check `<head>` in each page for:

- Render-blocking resources that should be deferred
- Preconnect hints for external origins used (fonts, CDNs)
- Correct order: charset > viewport > title > meta > canonical > OG > CSS > preloads

Add `<link rel="preconnect">` for any external font or CDN domains being used.

## PHASE 5: ACCESSIBILITY AUDIT

Run a thorough WCAG 2.1 AA checklist against every page.

**5.1 — Color contrast**
Identify every text/background color combination in use. Verify minimum contrast ratios:

- Normal text (under 18pt): 4.5:1
- Large text (18pt+ or 14pt+ bold): 3:1
- UI components and states: 3:1

If the current brand palette has any combinations that fail, flag them and propose specific adjusted values that preserve brand identity while meeting contrast requirements.

**5.2 — Interactive element accessibility**
Every interactive element must have:

- A visible focus indicator (not removed by `outline: none` without a custom replacement)
- An accessible name (via visible label, `aria-label`, or `aria-labelledby`)
- Keyboard operability (no mouse-only interactions)
- Correct ARIA roles where native HTML semantics are insufficient

Check every button, link, and form control. The navigation dropdown menus must be keyboard-navigable.

**5.3 — Form accessibility**
Check the contact form (contact/):

- Every input must have an associated `<label>` with matching `for`/`id` pairing
- Required fields must be marked with both visual indicator and `required` attribute and `aria-required="true"`
- Error states must use `aria-describedby` or `aria-live` region for screen reader announcements
- Form submission feedback must be accessible

**5.4 — Image accessibility**
Beyond alt text (covered in 4.1), check:

- Decorative images use `alt=""` and `role="presentation"`
- Complex images (charts, diagrams) have either a long description or `aria-describedby` pointing to a nearby text explanation
- No image used to convey text that isn’t replicated in actual text

**5.5 — Document language and landmarks**

- Verify `lang="en"` on every page’s `<html>` element
- Verify page has at minimum these ARIA landmarks: `banner` (header), `navigation` (nav), `main`, `contentinfo` (footer)
- If multiple nav elements exist, each needs a unique `aria-label`
