### ROY: Return on Your Words

Once the rendering cost collapsed, the real question became unavoidable: **What did the words actually buy?**

> **The ROY Formula**
Clarity Delivered ÷ Words Required to Deliver It
Return on Your Words = understanding extracted ÷ words invested
> 

A 47-bullet Slack message? Poor ROY. A “quick summary” that still requires a meeting to decode? Negative ROY. A diagram generated from 50 well-chosen words that a team grasps in five seconds and uses correctly? Potentially extraordinary ROY.

That is why I don’t believe a picture is automatically worth 1,000 words. Some pictures are decorative wallpaper with a superiority complex. Some diagrams are just confusion with borders and connector lines.

The real test is simpler: **Did the visual remove enough confusion to justify the effort it took to make it?**

> A picture is only worth the words it saves, the ambiguity it kills, and the momentum it creates.
> 

The first diagram usually fails that test. It wants to look finished. It wants to reassure you that the path is known, the thinking is done, and the complexity has been tamed. Real thinking is uglier than that.

Real thinking loops. It forks. It backtracks. It trims its own nonsense. It discovers, usually late, that the clever version and the useful version are not the same thing.

### Mermaid Is Not AI

That statement needs its own section because almost every first-time reader assumes otherwise.

When this experiment is described as "six AI platforms competing to generate the same Mermaid diagram," the natural inference is that Mermaid is an AI output format, something that only exists because a model produced it. That inference breaks the entire council comparison. So before the results, the format needs a proper introduction.

A Mermaid diagram is text.

Not an image. Not a render. Not a generative output. It is a structured syntax, a specific grammar written in plain characters, that a JavaScript library reads and converts into a visual diagram inside a browser. The diagram you see on screen is what the text decided to be. The text is the source file. The visual is the compiled result.

And here is the test that settles the question cleanly:

If a stubborn enough person can sit down with a plain text editor and type the structure by hand, the structure itself is not AI. It is text. It is a human-readable, machine-renderable artifact.

A Mermaid diagram passes that test.

So does HTML.

So does SVG.

So does CSS.

So does a task list in Notepad.

So does a rough draft in Markdown.

A human can write any of them. A machine can write any of them. The file format does not become AI simply because an AI is one possible author.

Here is what Mermaid looks like in its raw form:

```
flowchart LR
    A[Request Submitted] --> B{Approved?}
    B -- Yes --> C[Schedule Deployment]
    B -- No --> D[Return to Requester]
```

A human typed that. It renders into a clean flowchart. No model. No API. No subscription. A person who is stubborn enough and curious enough can write Mermaid code by hand in Notepad, paste it into the Mermaid Live Editor, and produce a publication-ready diagram without involving AI at any step.

Mermaid’s own documentation describes it as a JavaScript-based diagramming tool that uses Markdown-inspired text definitions to render diagrams. The Markdown-inspired part matters: the syntax is intentionally readable, intentionally learnable, and intentionally writable by a human without specialized tooling. That is a design principle, not an accident.

It also means a Mermaid diagram is not a locked visual artifact. It is source. You can inspect it. Revise it. Store it in source control. Diff it. Hand it to a human. Hand it to a machine. Hand it back and forth between both until the structure gets better.

That is a very different thing from saying an AI made me a picture.

### Mermaid the Syntax vs. [Mermaid.ai](http://Mermaid.ai) the Platform

This is where the terminology confusion compounds, because the name overlap is genuinely misleading.

Mermaid is the language.

[Mermaid.ai](http://Mermaid.ai) is a product built to help people generate Mermaid faster.

Mermaid is the underlying format.

[Mermaid.ai](http://Mermaid.ai) is one possible interface layered around that format.

Mermaid is the grammar.

[Mermaid.ai](http://Mermaid.ai) is one authoring environment.

Mermaid the syntax is the open-source diagramming language and renderer. It is a JavaScript library. It has a specification. It has a community. It runs in browsers, in documentation tools, in markdown renderers, and in platforms like GitHub, Notion, and Microsoft Loop that have adopted it natively. The language itself has no AI in it.

[Mermaid.ai](http://Mermaid.ai) the platform is a separate product: a generative AI interface built around the Mermaid ecosystem. It is purpose-built to help users produce Mermaid diagrams faster using AI. It is useful. It is well-designed for what it does. But it is a tool layered on top of the language, not the language itself.

ChatGPT, GitHub Copilot, Microsoft Copilot, Perplexity, Claude, and a growing list of extensions in Mermaid’s official documentation can all generate valid Mermaid syntax on request. None of them invented the format. They are all authoring tools for a language that exists independently of them.

> A Mermaid diagram is not AI. It is text. The same way an HTML file is not AI, a spreadsheet is not AI, and a list of tasks in Notepad is not AI, unless a machine wrote it. And even then, the format itself is neutral.
> 

This distinction matters because the Council of AIs was not a beauty contest for images. It was a test of authoring. Which AI could produce the most accurate, most maintainable, most structurally sound Mermaid code from a complex evolving brief? The diagram is the visible output. The code is the work product. The AI is the author under evaluation, the same way you would evaluate which developer writes the cleanest SQL.

Once you understand that framing, the most unexpected result of the experiment stops sounding strange.

### The One That Should Have Won

[Mermaid.ai](http://Mermaid.ai) was the obvious ringer.

The council was assembled to generate Mermaid diagrams. [Mermaid.ai](http://Mermaid.ai) exists to generate Mermaid diagrams. If the bet had been placed before the experiment started, [Mermaid.ai](http://Mermaid.ai) would have looked like the safest money on the table.

It did not make the council.

Not because it was bad at drawing. Because it was too eager to draw.

#### The Failure Mode

The failure has a name: premature rendering.

[Mermaid.ai](http://Mermaid.ai) is architecturally optimized for immediate execution. The moment it detects diagram intent, it produces output. There is no observable hold state, no way to feed context across multiple turns and tell the platform to wait before generating. The council methodology depends on layering: narrative first, system logic second, relationship mapping third, draw instruction last. [Mermaid.ai](http://Mermaid.ai) collapsed those phases and started producing on the first signal.

The diagrams were not inaccurate. They were premature. Built on an incomplete read of a brief that was still being written.

Speed is not the same thing as synthesis. In quick-turn drafting, speed is valuable. In serious multi-turn reasoning, speed without patience can produce a cleaner lie faster.

That is exactly the kind of thing this article is arguing against.

The first diagram usually lies not because the author is malicious, but because the author commits before the structure is honest. [Mermaid.ai](http://Mermaid.ai), in this test, reproduced that same problem at the platform level.

The drawing specialist missed the brief.

#### The Recommendation

This is not a hit piece, and it is not a complaint disguised as one.

It is a product recommendation.

If the Mermaid team ever wants to make the platform stronger for serious architecture, systems, and recursive ideation work, the next feature is not "draw faster." It is "wait better."

Build a Think mode. A context-intake phase where users can feed narrative, files, markdown, ambiguity, and system logic across multiple turns without triggering generation. Let the model accumulate meaning before it commits to form. Then switch into draw mode.

Hold.

Hold.

Hold.

Go.

The council was built to draw.

The specialist built for drawing did not make the council because it could not hold long enough to draw well.

Sometimes the smartest move in diagramming is not to render faster.

It is to hold a little longer until the structure stops lying.
