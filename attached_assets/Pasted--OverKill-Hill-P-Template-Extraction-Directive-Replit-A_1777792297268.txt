## STEP 3: NAMING CONVENTION

Every template file must follow this exact naming convention:

```
[section]-[subsection]-template.html
```

Examples:

- Homepage → `home-template.html`
- Writings Hub → `writings-hub-template.html`
- Individual Article → `writings-article-template.html`
- Projects Hub → `projects-hub-template.html`
- Individual Project → `projects-project-template.html`
- About → `about-template.html`
- Contact → `contact-template.html`
- Legal → `legal-template.html`
- Manifesto → `manifesto-template.html`
- Universe → `universe-template.html`
- Prompt Forge → `prompt-forge-template.html`
- Found-ry → `found-ry-template.html`
- 404 → `404-template.html`
- Under Construction → `under-construction-template.html`

**Page type consolidation rule:** If two or more pages share the same structural layout (same section regions, same component arrangement), they share one template. For example: “First Diagram Is a Liar,” “Biases as Constants,” and “Magnus Saga” likely all use the same article layout — they produce one shared template called `writings-article-template.html`, not three separate files. Use judgment to identify which pages are layout-identical and consolidate them. Document your consolidation decisions in the template header comment.

## STEP 4: TEMPLATE CONSTRUCTION RULES

For each page that lacks a template, create the template file according to these rules:

### 4.1 — What stays in the template

Keep all structural and chrome elements exactly as they appear in the source page:

- The full `<head>` block, with all CSS links, meta tags, and scripts intact
- The complete site `<header>` and `<nav>` (navigation must be fully functional — the template is a working HTML file, not a wireframe)
- The complete site `<footer>` with all links and social icons
- The skip-to-content link
- Every layout container, wrapper div, section, and grid structure
- CSS class names on every element (so the template reflects the actual layout grid)
- Any JavaScript includes or inline scripts that drive layout behavior (hamburger menu, dropdowns, etc.)
- The announcement/hot-forge banner at the top if present on the source page

### 4.2 — What gets stripped

Remove all page-specific content, replacing it with labeled placeholder comments:

|Content type                           |Replace with                                                                             |
|---------------------------------------|-----------------------------------------------------------------------------------------|
|Page `<title>` text                    |`[PAGE TITLE]                                                                            |
|`<meta name="description">` content    |`[PAGE META DESCRIPTION — 120-160 chars]`                                                |
|`og:title` content                     |`[OG TITLE]`                                                                             |
|`og:description` content               |`[OG DESCRIPTION]`                                                                       |
|`og:url` content                       |`[CANONICAL URL]`                                                                        |
|`og:image` content                     |`[OG IMAGE PATH]`                                                                        |
|`twitter:title` / `twitter:description`|`[TWITTER TITLE]` / `[TWITTER DESCRIPTION]`                                              |
|`<link rel="canonical">` href          |`[CANONICAL URL]`                                                                        |
|Page `<h1>` text                       |`[PAGE HEADLINE]`                                                                        |
|Any `<h2>` section heading             |`[SECTION HEADING]`                                                                      |
|Any `<h3>` subsection heading          |`[SUBSECTION HEADING]`                                                                   |
|Body paragraph text                    |`<!-- [BODY CONTENT] -->` (preserve the `<p>` tag, empty the text)                       |
|Article body content                   |Single `<p>` containing `[ARTICLE BODY CONTENT]` inside the content container            |
|Image `src` attributes                 |`/assets/img/[IMAGE-FILENAME.ext]`                                                       |
|Image `alt` attributes                 |`[Descriptive alt text for image]`                                                       |
|Inline `href` links within body content|`[LINK URL]`                                                                             |
|Any hardcoded dates                    |`[PUBLICATION DATE]`                                                                     |
|Author bylines                         |`[AUTHOR NAME]`                                                                          |
|Read-time estimates                    |`[X min read]`                                                                           |
|Any version badges (e.g., v0.3)        |`[VERSION BADGE]`                                                                        |
|JSON-LD structured data values         |Placeholder strings matching the key names                                               |
|Breadcrumb page-specific items         |`[CURRENT PAGE NAME]` / `[CURRENT PAGE URL]`                                             |
|Ko-fi or CTA section content           |Preserve the CTA container structure; replace copy with `[CTA HEADLINE]` and `[CTA BODY]`|

### 4.3 — What the template must NOT do

- Do not alter the nav links — they must remain live and accurate (the nav is chrome, not content)
- Do not remove the footer social links — they are structural chrome
- Do not change CSS classes or IDs on any element
- Do not remove any structural `div`, `section`, `article`, or `aside` wrappers — even if empty after stripping
- Do not convert the template to a component, partial, or include format — it must be a complete, self-contained, valid HTML file that renders correctly in a browser

### 4.4 — Template header comment block

Every template file must open (after `<!DOCTYPE html>`) with this comment block before the `<html>` tag:

```html
<!--
  =====================================================
  OverKill Hill P³™ — Page Template
  =====================================================
  Template:       [template filename, e.g., writings-article-template.html]
  Covers pages:   [list every source page this template was derived from]
  Derived from:   [primary source file path, e.g., /writings/first-diagram-is-a-liar/index.html]
  Created:        [date created, e.g., May 2026]
  Last updated:   [same as created on initial generation]

  PURPOSE:
  Stripped structural clone of the above page(s). All page-specific
  content replaced with labeled placeholders. Navigation, footer,
  CSS, and JS are live and functional. Use this file as the starting
  point when creating a new page of this layout type.

  USAGE:
  1. Copy this file to the appropriate directory path
  2. Rename to index.html
  3. Replace all [PLACEHOLDER] tokens with real content
  4. Update <head> meta, canonical URL, and OG tags
  5. Commit and deploy

  Consolidated from: [Y / N — if Y, list which pages share this layout]
  =====================================================
-->
```

## STEP 6: VALIDATION

After creating all templates:

1. Verify every template file opens without HTML parse errors
1. Verify every template file has the correct header comment block
1. Verify the `/assets/templates/README.md` lists every template created
1. Verify no template contains any page-specific proper nouns, URLs, dates, or body text that should have been stripped
1. Verify every template’s nav and footer links are intact and point to live site URLs (not relative paths that would break from the `/assets/templates/` subdirectory — use root-relative or absolute paths for all nav/footer links)

**Critical on point 5:** Because templates live at `/assets/templates/filename.html` rather than at a page root, any relative CSS, JS, or image paths that worked at page root level will break. All asset paths in templates must be root-relative (starting with `/`) or absolute (`https://overkillhill.com/`). Audit and fix every `href`, `src`, and `url()` in template files to ensure nothing uses a relative path that would resolve incorrectly from the `/assets/templates/` directory depth.
