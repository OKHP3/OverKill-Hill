## DOMAIN 1: 2026 WEB BEST PRACTICES

Audit every HTML file against the following 2026-era baseline. Flag each violation. Implement every fix that can be made without risk of visual or functional regression.

### 1.1 — Document-level requirements

Every page must have:

| Requirement | Correct Implementation |
|---|---|
| DOCTYPE | `<!DOCTYPE html>` — no XHTML, no legacy variants |
| HTML lang | `<html lang="en">` on every page |
| Charset | `<meta charset="UTF-8">` as the **first** element inside `<head>` |
| Viewport | `<meta name="viewport" content="width=device-width, initial-scale=1">` |
| Color scheme | `<meta name="color-scheme" content="dark light">` |
| Theme color | `<meta name="theme-color" content="#2a2320">` (brand dark) |
| IE compat | No `X-UA-Compatible` meta — this is dead in 2026, remove it if present |
| `<main>` element | One per page, with `id="main"` so skip-link targets are consistent |
| Skip link | `<a class="skip-link" href="#main">Skip to main content</a>` as the first child of `<body>` — text and anchor must be consistent across all pages |

Check: The live site shows `href="#main"` on some pages and `href="#content"` on others. Standardize to `href="#main"` and ensure every page's main content container has `id="main"`.

### 1.2 — Resource loading best practices

For every `<link>`, `<script>`, and `<img>` tag across all pages:

**CSS:**
- All stylesheets loaded via `<link rel="stylesheet">` in `<head>`
- No `@import` inside CSS files that chains blocking requests
- Add `<link rel="preload" as="style">` for the primary stylesheet if not already present

**JavaScript:**
- All `<script>` tags referencing external files must have `defer` attribute unless the script must execute synchronously (almost never the case for a static site)
- No `<script>` tags without `defer` or `async` unless there is a documented reason — comment that reason inline
- No `<script>` tags placed in `<head>` without `defer`

**Images:**
- `loading="lazy"` on all images that appear below the fold
- `loading="eager"` and `fetchpriority="high"` on the primary hero image of each page
- `decoding="async"` on all non-hero images
- Explicit `width` and `height` attributes on every `<img>` to prevent Cumulative Layout Shift

**External domains:**
- Add `<link rel="preconnect">` for every external origin used (Google Fonts, Google Analytics, any CDN)
- Add `<link rel="dns-prefetch">` as a fallback for browsers that don't support preconnect

### 1.3 — Deprecated patterns to eliminate

Scan all HTML files and remove or replace:

| Pattern | Action |
|---|---|
| `<meta http-equiv="X-UA-Compatible">` | Remove entirely |
| `border="0"` on `<img>` | Remove — use CSS |
| `align=""` attribute on any element | Remove — use CSS |
| `onclick=""` inline handlers | Move to external JS file |
| `style=""` inline styles | Move to external CSS (see Domain 6) |
| `<b>` and `<i>` for emphasis | Replace with `<strong>` and `<em>` |
| `target="_blank"` without `rel="noopener noreferrer"` | Add the rel attribute to every external link |
| `javascript:void(0)` hrefs | Replace with proper button elements or real URLs |
| HTML comments containing development notes | Remove from production files |

### 1.4 — Modern HTML features to add where missing

- `<meta name="referrer" content="strict-origin-when-cross-origin">` in every `<head>`
- `autocomplete` attributes on all form inputs (contact form)
- `inputmode` attributes on numeric or email inputs
- `<time datetime="[ISO date]">` wrapping any dates displayed in article bylines

## DOMAIN 3: GOOGLE ANALYTICS

### 3.1 — Presence check

Scan every HTML file for the Google Analytics or Google Tag Manager snippet. Check for:
- A `<script>` tag loading `gtag.js` or `analytics.js`
- A Google Tag Manager `<noscript>` block in `<body>`
- Any `G-XXXXXXXXXX` or `GTM-XXXXXXX` measurement ID present

Document: Is analytics present? On which pages? On all pages or only some?

### 3.2 — Consistency

Analytics tracking must be present on **every** page without exception, including:
- `404.html` (tracking 404 hits is valuable — this page needs the snippet)
- `under-construction.html` (you want to know how often users land on stub pages)
- Every writing, project, legal, and about page

If any page is missing the snippet, add it. Copy the snippet exactly from a page where it is currently correct.

### 3.3 — Placement

The Google Tag Manager snippet has two parts:
- The `<script>` block goes in `<head>` as high as possible (after charset and viewport, before other scripts)
- The `<noscript>` iframe goes immediately after the opening `<body>` tag

Verify this placement is correct on every page. An analytics snippet at the bottom of `<body>` will miss users who bounce before the page finishes loading.

### 3.4 — No duplicate snippets

Verify no page loads the analytics snippet more than once. A double-fire inflates session counts.

### 3.5 — Event tracking check

Document whether any custom events are tracked (button clicks, outbound links, form submissions, Ko-fi clicks, Fiverr clicks). If not, note this as a follow-up opportunity — do not implement custom events in this audit without explicit instruction. Flag the absence and list the highest-value events to add in a future session:
- Ko-fi link clicks (conversion signal)
- Fiverr link clicks (conversion signal)
- Article completion scroll depth (engagement signal)
- Contact form submission (conversion signal)
- Outbound links to glee-fully.tools and askjamie.bot

### 3.6 — Measurement ID documentation

Extract and document the current Measurement ID (format: `G-XXXXXXXXXX`) from the source files. Record it in a comment at the top of the analytics snippet on each page so it's visible at a glance during future maintenance. Format:

```html

## DOMAIN 7: CSS AND JS ASSET EXTERNALIZATION

This is a hard rule. No page should have styles or scripts defined or loaded anywhere except the `/assets/` folder. Inline styles and inline scripts are maintenance traps and SEO liabilities.

### 7.1 — Inline style audit

Scan every HTML file for `style=""` attributes on any element. For each one found:

1. Document: which file, which element, what styles are applied
2. Determine: does an existing CSS class already handle this? If yes, replace the inline style with the class
3. If no existing class handles it: create a named utility class in the appropriate stylesheet and apply it

Do not leave a single `style=""` attribute in any HTML file. Convert all of them.

### 7.2 — Inline script audit

Scan every HTML file for `<script>` blocks that contain JavaScript code directly (not `<script src="...">`). For each one found:

1. Document: which file, what the script does
2. Determine: does this script belong in an existing JS file in `/assets/js/`?
3. Move the JavaScript to the appropriate file in `/assets/js/`
4. Replace the inline `<script>` block with `<script src="/assets/js/[filename].js" defer></script>`

Exception: Google Analytics / Google Tag Manager snippets are expected to be inline `<script>` blocks. These do not need to be externalized. Leave them in place.

Exception: JSON-LD structured data `<script type="application/ld+json">` blocks are not JavaScript and must remain inline in `<head>`. Do not move these.

### 7.3 — `<style>` block audit

Scan every HTML file for `<style>` blocks in `<head>` (not external stylesheet links — actual inline `<style>` blocks containing CSS rules). For each one found:

1. Document: which file, what rules it contains
2. If the styles are page-specific and don't exist anywhere in `/assets/css/`: add them to the most appropriate external stylesheet (or create a page-specific CSS file in `/assets/css/pages/` if the rules are genuinely unique to that one page)
3. Remove the `<style>` block from the HTML file
4. Ensure the external stylesheet is already linked in the page's `<head>`

Exception: Critical CSS that is intentionally inlined for render performance is acceptable **only** if it is under 1KB and is duplicated in the external stylesheet. Document any such exceptions with a comment.

### 7.4 — Asset folder structure verification

After externalization, verify `/assets/` is organized as follows (create any missing directories):

```
/assets/
  /css/
    main.css          ← primary global stylesheet
    /pages/           ← page-specific overrides (only if truly needed)
  /js/
    main.js           ← primary global script (nav, etc.)
    /pages/           ← page-specific scripts (only if truly needed)
